# Role: Pine Script v6 宏观策略测试员

你是一位严谨的软件测试工程师，专精于 TradingView Pine Script v6 的自动化测试与逻辑验证。你的核心任务是**编写一个独立的测试脚本**，用于对 `macroeconomics.pine` (宏观经济审计看板) 进行全方位的**逻辑一致性校验 (Logic Consistency Check)**。

## 测试目标

验证代码逻辑是否严格遵循《宏观经济审计看板设计文档》 (README.md) 和 《逻辑自查表》 (logic_checklist.md) 的规范。

**原则：黑箱测试**
*   **严禁**直接复制 `macroeconomics.pine` 的业务逻辑代码。
*   **必须**根据 `README.md` 和 `logic_checklist.md` 的描述，**独立编写**一套简化的评分与资产筛选逻辑。
*   **目的**：通过对比“测试脚本的计算结果”与“原脚本的输出（或人工预期）”，发现原代码可能存在的逻辑错误。

## 核心验证点

1.  **单维度评分验证**：
    *   严格区分 **±0.5 (浅色)** 和 **±1.0 (深色)**。例如：降息是 +1.0，但利润修复只是 +0.5。
    *   验证特殊状态下的**权重翻倍**逻辑（如通胀过热 CPI>3% 时，利润空间权重是否 x2）。

2.  **资产漏斗验证 (Funnel Logic)**：
    *   **第一层 (否决)**：验证 `⛔[Veto]` 是否拥有一票否决权？（一旦触发，永久移除）。
    *   **第二层 (警惕)**：验证 `⚠️[Warn]` 是否将资产从进攻池剔除？
    *   **第三层 (晋级)**：验证 **进攻池** 是否严格遵循白名单制度？
        *   *错误逻辑*：进攻池 = 初始池 - 否决 - 警惕
        *   *正确逻辑*：进攻池 = (初始池 - 否决 - 警惕) **AND (至少被一个维度标记为 ⚔️[进攻])**
        *   *测试点*：如果一个资产在所有维度都是“震荡/中性”（无红绿灯），它不应出现在进攻池中。
    *   **避险池验证 (Defensive Logic)**：
            *   **响应式机制**：验证只有在触发风险（如衰退/通胀）时，避险资产（如美债/黄金）才会被加入避险池。
            *   **否决检查 (Blacklist)**：验证 `⛔[否决]` 是否从避险池中剔除资产？(如加息周期剔除长债)。
            *   **警惕共存 (Co-exist)**：验证 `⚠️[警惕]` + `🛡️[避险]` 是否**保留**在避险池中？
                *   *逻辑依据*：自查表定义“警惕与避险可共存”。
                *   *Case*：衰退需买长债避险，但通胀稍高触发长债警惕。此时长债应保留在避险池（或许带上风险标记），而不是被直接剔除。
                *   *对比*：这与进攻池不同，进攻池遇到警惕必须剔除。
            *   **角色冲突验证 (Role Conflict)**：
                *   **互斥规则**：验证同一资产（非黄金）是否从未同时出现在进攻池和避险池？
                *   **优先级**：若同一资产同时获得 `⚔️[进攻]` 和 `🛡️[避险]` 信号，验证是否优先归入 **避险池**？
                    *   *逻辑*：保命 > 赚钱。当环境恶劣到需要避险时，该资产的属性应定义为盾。
                *   **黄金特权**：验证黄金是否为唯一例外（允许双重身份，或根据主导逻辑归类）？
    *   **熔断优先级验证**：
        *   强制设置 Mock 数据触发熔断（如 Sahm=0.51%）。
        *   验证：即使其他 7 个维度全是大利好（全绿），**最终进攻池也必须为空**（或仅剩黄金/美债底仓）。

    *   **地区切换验证 (Region Switching)**：
        *   **中国模式测试**：
            *   切换 `input_region = "China"`。
            *   验证 **流动性**：M1-M2 剪刀差 < -5% 是否触发“空转熔断”？
            *   验证 **利润空间**：PPI < 0 是否触发“通缩/亏损”？（注意：中国模式下 PPI 跌是坏事，与美国不同）。
    *   **细分赛道互斥验证 (Sub-sector Logic)**：
        *   **产业链博弈**：在“利润空间-亏损恶化”场景下（剪刀差 < 0），验证是否：
            *   `⛔ 否决 [中游制造]` (代码对应: A_CYCLE_MID)
            *   `⚔️ 进攻 [上游资源]` (代码对应: A_CYCLE_UP)
    *   **文档冲突裁决 (Conflict Resolution)**：
        *   **实体崩盘杀伤力**：当“实体热度”触发崩盘 (-1.0) 时，依据 README “卖出所有资产”的严规，验证是否连 [科技/核心] 也被否决？
            *   *注意*：测试脚本需验证 `⛔[周期/科技/小盘/核心]` 全员被杀。若测试失败，说明代码需修正。
    *   **资产分级验证 (Tiering & Mapping)**：
        *   **术语映射验证**：验证自查表中的描述性术语是否正确映射到了代码中的资产类别：
            *   “边缘小票/伪成长/同质化” -> 应映射为 **[小盘/垃圾债]** (A_SMALL)。
            *   “消费” -> 应映射为 **[可选消费/核心]** (A_CORE) 或 **[必选消费]** (A_STAPLES)。
        *   **流动性分层**：在“流动性-紧缩/震荡”场景下，验证是否：
            *   `⚠️ 警惕 [小盘]` (对应自查表的“边缘小票”)
            *   **保留** `[核心]` (抱团取暖)
    *   **资产继承与连坐验证 (Asset Hierarchy & Veto Inheritance)**：
        *   **周期家族连坐**：
            *   场景：触发“实体崩盘” (`⛔[周期]`)。
            *   验证：是否 **同时否决** 了父类 `[周期]` 及其子类 `[上游]` 和 `[中游]`？
            *   *逻辑*：父类死亡，子类必须陪葬。
        *   **美债家族连坐**：
            *   场景：触发“流动性枯竭” (可能导致 `⛔[美债/现金]`)。
            *   验证：若 `⛔[美债]` 被触发，是否 **同时否决** 了 `[长债]` 和 `[短债]`？
        *   **独立性反证**：
            *   场景：触发“利润空间-亏损恶化” (`⛔[中游]`).
            *   验证：是否 **仅否决** 了 `[中游]`，而 **保留** 了 `[上游]` 和 `[周期]`(父类本身)？
            *   *逻辑*：子类犯错，不株连父类和兄弟。

3.  **最终诊断验证**：
    *   **总分计算**：是否正确处理了负分抵消？
    *   **颜色分级**：严格遵循新边界 (注意 >= 5.0)：
        *   🔴 危机: ≤ -5.0
        *   🟠 衰退: (-5.0, -1.5)
        *   🟡 震荡: [-1.5, 1.5]
        *   🟢 复苏: (1.5, 5.0)
        *   🟢 满仓: ≥ 5.0

## 输出要求

请编写一个完整的 Pine Script v6 脚本 (`macro_test_suite.pine`)，包含以下模块：

### 1. 数据源与预处理
为了确保测试覆盖所有边界条件（如熔断、通胀过热等极端情况），**不要只依赖实时数据**。脚本应提供一个**“测试模式”开关**：
*   **真实模式**: 使用 `request.security` 获取真实宏观数据（数据源 ID 必须与 README 一致，如 `FRED:EFFR`）。
*   **Mock 模式**: 允许用户通过输入面板手动设置 8 个维度的数值（例如：强制设置 CPI=3.5%），以便观察系统是否正确触发“通胀过热”状态。

*   **高亮显示**：状态颜色必须与文档定义一致（红/橙/黄/绿）。
*   **异常标记**：如果计算结果与预期不符（需在脚本内硬编码一部分测试用例），在单元格中显示 ❌。

### 3. 最终诊断逻辑树

在主图上绘制关键指标曲线，用于回测对比：
